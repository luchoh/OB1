# PRD: OB1 Graph Augmentation

Date: 2026-03-17
Status: Proposed
Owner: Platform / Retrieval / Knowledge Systems

## Summary

Add a graph layer to Open Brain Local using the existing `neo4j` service, but keep PostgreSQL + `pgvector` as the canonical memory store.

The graph is a derived projection, not a new source of truth.

Its purpose is to add:

- explicit relationships
- provenance and lineage
- multi-hop traversal
- graph-aware retrieval expansion

The graph must improve the brain's ability to answer questions like:

- what documents, emails, and thoughts are connected to this property?
- what people, projects, and systems keep co-occurring?
- why does this answer appear related?
- what else is one hop away from this memory?

## Problem

The current local brain is strong at:

- capture
- semantic retrieval\n- grounded answering over retrieved evidence\n\nIt is weaker at:

- explicit relationship queries\n- provenance navigation across sources\n- multi-hop reasoning across people, projects, devices, documents, and derived thoughts\n- controlled context expansion beyond vector nearest-neighbor search\n\nExample failure shape:

- vector retrieval can find the right attachment or thought\n- but it cannot natively express that:\n  - an email has attachments\n  - an attachment produced a distilled summary\n  - a person appears in the email and also in a project\n  - a property is referenced by multiple artifacts\n\nThe repo already assumes relational links and provenance matter. It does not yet have a first-class graph projection.\n\n## Goals\n\n- Add Neo4j as a relationship layer without replacing PostgreSQL.\n- Make graph data traceable back to canonical OB1 rows.\n- Start with high-confidence provenance and metadata links before open-ended knowledge-graph extraction.\n- Support graph-assisted retrieval and debugging of why a result was surfaced.\n- Keep all extraction local-only and consistent with the current MLX/Docling/Postgres stack.\n\n## Non-Goals\n\n- Replacing PostgreSQL or `pgvector` as the main memory store\n- Making Neo4j the primary write path\n- Building an unconstrained \"graph of everything\"\n- Letting low-confidence LLM entity extraction silently reshape the graph\n- Migrating all retrieval to graph queries in v1\n\n## Product Position\n\nOpen Brain remains:\n\n- canonical storage in PostgreSQL\n- canonical embeddings in `thoughts.embedding`\n- canonical grounded answering over stored evidence\n\nNeo4j becomes:\n\n- a derived context and relationship index\n- a provenance explorer\n- a controlled context-expansion layer for retrieval\n\n## Current-State Assumptions\n\nFrom the current local stack:\n\n- PostgreSQL `ob1` is canonical\n- `mlx-server` is canonical inference\n- `ob1-embedding` is canonical embeddings\n- `docling` is canonical document extraction\n- `neo4j` is already present in the environment as a healthy internal service\n- service discovery should happen through Consul, not hardcoded IPs\n\n## Research-Backed Findings\n\nThis PRD is grounded in [12-graph-augmentation-evidence.md](/Users/luchoh/Dev/OB1/docs/12-graph-augmentation-evidence.md#L1).\n\nThe most important architectural facts are:\n\n- the local Neo4j Desktop setup includes Enterprise-capable features for this single-machine development environment\n- multiple Neo4j databases are therefore a real available capability on this machine\n- transactions cannot span multiple Neo4j databases\n- Neo4j has native vector indexes for both nodes and relationships\n- Neo4j vector indexes consume operating system filesystem-cache memory in addition to normal Neo4j memory planning\n- official Neo4j GraphRAG tooling now exists as a first-party Python package\n- pgvector already provides exact search, ANN, filtering, and hybrid search in the canonical PostgreSQL store\n\nDesign consequence:\n\n- Neo4j should enter OB1 first as a derived relationship layer\n- PostgreSQL + pgvector should remain the canonical vector and memory layer in v1\n\n## Core Decision\n\n### Source of Truth\n\nPostgreSQL remains the source of truth for:\n\n- `thoughts`\n- importer outputs\n- metadata\n- embeddings\n- answer provenance\n\n### Graph Role\n\nNeo4j stores a derived projection built from PostgreSQL rows and selected source metadata.\n\nThe projection must always support:\n\n- rebuild from canonical state\n- incremental sync\n- provenance back to source rows\n\n### Sync Style\n\nUse an asynchronous projector.\n\nDo not make `capture_thought` or importer writes block on Neo4j.\n\nWrite flow:\n\n1. importer or MCP tool writes canonical row to PostgreSQL\n2. graph projector observes new/changed rows\n3. projector upserts graph nodes and relationships in Neo4j\n\n### Local Neo4j Database Strategy\n\nBecause the local Desktop setup supports multiple databases on this machine, the default database layout should be:\n\n- one production graph database, recommended name: `ob1_graph`\n- one staging graph database for projector and schema experiments, recommended name: `ob1_graph_stage`\n\nDo not default to:\n\n- one graph database per source\n- one graph database per import job\n- cross-database write choreography\n\nReason:\n\n- multiple databases are available\n- but transactions cannot span them\n- so broad database fragmentation adds complexity faster than it adds value\n- explicit production/staging names reduce the chance of accidental cross-environment writes\n\n## Scope\n\n### Phase 1: Provenance Graph\n\nBuild only high-confidence graph structure from existing data.\n\nThis phase should not depend on fuzzy entity extraction.\n\nInitial nodes:\n\n- `Thought`\n- `Conversation`\n- `Email`\n- `Attachment`\n- `Document`\n- `DictationArtifact`\n\nInitial relationships:\n\n- `DERIVED_FROM`\n- `PART_OF`\n- `HAS_ATTACHMENT`\n- `SUMMARIZED_AS`\n- `DISTILLED_TO`\n- `REFERENCES_SOURCE`\n\nPrimary value:\n\n- answer explainability\n- traversal across related artifacts\n- source lineage for distilled rows\n\n### Phase 2: Typed Entity Graph\n\nAdd extracted entities from selected high-value rows.\n\nInitial entity labels:\n\n- `Person`\n- `Organization`\n- `Project`\n- `Device`\n- `Place`\n- `Property`\n- `Concept`\n\nInitial relationship types:\n\n- `MENTIONS`\n- `ABOUT`\n- `USES`\n- `LOCATED_AT`\n- `OWNED_BY`\n- `SENT_BY`\n- `ASSOCIATED_WITH`\n- `RELATED_TO`\n\nThis phase must only operate on:\n\n- distilled rows\n- document summaries\n- email thoughts\n- dictation thoughts\n\nDo not start with raw chunk rows.\n\n### Phase 3: Hybrid Retrieval\n\nUse Neo4j to improve retrieval quality without replacing vector search.\n\nInitial pattern:\n\n1. semantic retrieval in PostgreSQL\n2. graph expansion by one or two hops\n3. rerank or package the neighborhood as additional evidence for `ask_brain`\n\nExample:\n\n- vector search finds a property summary\n- graph expansion finds related attachment, email, owner, and garage node\n- `ask_brain` answers from the expanded evidence neighborhood\n\n### Explicit Non-Goal for v1\n\nDo not make Neo4j a second canonical vector store in v1.\n\nNeo4j vector indexes are real and may become useful later, including for graph-local reranking or relationship-vector retrieval, but v1 should not duplicate canonical OB1 embeddings into Neo4j by default.\n\nThis prohibition applies to:\n\n- production write paths\n- background enrichment jobs\n- staging experiments that would be mistaken for canonical behavior\n\nReason:\n\n- pgvector already covers the canonical vector role\n- Neo4j vector search is approximate\n- Neo4j vector indexes add real OS-memory overhead\n- duplicating vectors in both systems creates sync and evaluation complexity without proven product value yet\n\n## Functional Requirements\n\n### FR1: Canonical Projection Identity\n\nEvery graph node derived from OB1 content must preserve a stable canonical identifier.\n\nMinimum required identity fields:\n\n- `thought_id` for `Thought`\n- source-specific identifiers when available\n- canonical source row references in node properties\n\nGraph identity must not depend on temporary file paths or traversal order.\n\n### FR2: Rebuildability\n\nThe graph must be fully rebuildable from PostgreSQL canonical rows plus source metadata.\n\nThe system must support:\n\n- full rebuild\n- incremental sync\n- targeted reproject of one source artifact or one conversation\n\n### FR3: Provenance Preservation\n\nEvery distilled graph node or entity relationship must preserve provenance.\n\nMinimum provenance fields:\n\n- source thought id\n- source type\n- source conversation or artifact id when available\n- extraction mode\n- extraction timestamp\n\n### FR4: Confidence Discipline\n\nThe graph must distinguish:\n\n- explicit metadata-derived links\n- deterministic provenance links\n- LLM-extracted entity links\n\nLLM-derived relationships must carry:\n\n- `extraction_method`\n- `confidence`\n- `evidence_text` or an equivalent evidence pointer\n\nThe system must never silently present low-confidence inferred relations as canonical facts.\n\n### FR5: Local-Only Extraction\n\nAll graph extraction and sync must use the local stack:\n\n- `mlx-server`\n- `docling`\n- PostgreSQL\n- Neo4j\n\nOptional future implementation references may draw from the official `neo4j-graphrag` Python package, but that package does not redefine the v1 architecture by itself.\n\nNo external graph enrichment service is allowed.\n\n### FR6: Query Surface\n\nThe graph layer must support at least these product capabilities:\n\n- neighbors of a thought\n- why two nodes are connected\n- all artifacts derived from one source\n- all source artifacts that support a distilled memory\n- graph-assisted expansion for `ask_brain`\n\n## Data Model\n\n### Canonical Node Labels\n\nPhase 1:\n\n- `Thought`\n- `Conversation`\n- `Email`\n- `Attachment`\n- `Document`\n- `DictationArtifact`\n\nPhase 2:\n\n- `Person`\n- `Organization`\n- `Project`\n- `Device`\n- `Place`\n- `Property`\n- `Concept`\n\n### Canonical Relationships\n\nPhase 1:\n\n- `DERIVED_FROM`\n- `PART_OF`\n- `HAS_ATTACHMENT`\n- `SUMMARIZED_AS`\n- `DISTILLED_TO`\n- `REFERENCES_SOURCE`\n\nPhase 2:\n\n- `MENTIONS`\n- `ABOUT`\n- `USES`\n- `LOCATED_AT`\n- `OWNED_BY`\n- `SENT_BY`\n- `ASSOCIATED_WITH`\n- `RELATED_TO`\n\n### Required Node Properties\n\nFor derived content nodes:\n\n- `canonical_id`\n- `source_system`\n- `source_type`\n- `title` when available\n- `created_at`\n- `updated_at`\n\nFor entity nodes:\n\n- `canonical_name`\n- `entity_type`\n- normalized identifiers when available\n\nFor all extracted nodes and edges:\n\n- `extraction_method`\n- `confidence`\n- `evidence_text` or evidence pointer\n\n## Sync Architecture\n\n### Projector\n\nAdd a graph projector service or script that:\n\n- reads canonical rows from PostgreSQL\n- computes graph nodes and edges\n- upserts them into Neo4j\n- records projection state back in PostgreSQL or a local sync table\n\nThe projector must support:\n\n- single-row projection\n- batch projection\n- full rebuild\n\n### Suggested Projection State\n\nTrack projection state in PostgreSQL, not only in Neo4j.\n\nSuggested fields:\n\n- canonical row id\n- projected revision hash\n- last projected at\n- last projection status\n- last projection error\n\nThis allows repair without guessing what Neo4j currently contains.\nIt also makes the graph explicitly disposable: the production graph must always be reconstructible from PostgreSQL canonical state plus deterministic source metadata.\n\n## Retrieval Integration\n\n### Default Retrieval Strategy\n\nKeep current default:\n\n1. PostgreSQL semantic retrieval\n2. distilled-first result policy\n3. grounded answer synthesis\n\n### Graph-Assisted Mode\n\nAdd a graph-assisted mode for `ask_brain`:\n\n1. retrieve top semantic matches\n2. collect one-hop or two-hop neighborhood in Neo4j\n3. filter neighborhood by relationship type and confidence\n4. pass the compact expanded evidence set to grounded answering\n\nThis mode must remain optional until evaluated.\n\n## MCP / API Surface\n\nPotential new tools after the projector exists:\n\n- `graph_neighbors`\n- `why_connected`\n- `source_lineage`\n- `expand_context`\n\nPotential `ask_brain` enhancement:\n\n- `graph_assisted=true`\n\nDo not expose graph mutation tools to clients in v1.\n\n## Evaluation\n\n### Graph Projection Evaluation\n\nMinimum checks:\n\n- every distilled row with a known source has a provenance edge\n- email rows connect to attachments\n- attachment-derived summaries connect back to their attachment\n- duplicate projector runs do not create duplicate nodes\n\n### Product Evaluation\n\nUse a fixed question set where graph should help:\n\n- what else is connected to this apartment?\n- which email produced this property summary?\n- what attachments support this answer?\n- what systems and projects are connected to this UniFi/Home Assistant setup?\n\nSuccess means:\n\n- better source traceability\n- better multi-hop recall\n- no loss of grounded-answer discipline\n\n## Rollout Plan\n\n### Phase 1\n\n- define graph schema\n- build provenance-only projector\n- support rebuild and incremental sync\n- validate on email + attachment + document imports\n\n### Phase 2\n\n- add entity extraction for distilled rows only\n- add confidence and evidence properties\n- validate on selected real corpora\n\n### Phase 3\n\n- integrate graph-assisted retrieval into `ask_brain`\n- run side-by-side evaluation against vector-only answering\n\n### Phase 4: Optional Neo4j Vector Evaluation\n\nOnly after Phases 1-3 prove useful:\n\n- evaluate whether a limited Neo4j vector index adds value for graph-local reranking or relationship-vector queries\n- compare that directly against the existing Postgres + pgvector path\n- do not duplicate canonical embeddings into Neo4j before this evaluation gate is passed\n- do not promote Neo4j vector storage to canonical unless it wins on a fixed evaluation set and the operational cost is justified\n\n## Risks\n\n- low-quality entity extraction can create persuasive but false graph structure\n- uncontrolled graph expansion can pollute answer context\n- duplicate identities can fragment the graph if normalization is weak\n- graph sync can drift unless rebuild and projection-state tools are explicit\n\n## Open Questions\n\n- Should graph projection state live in a dedicated Postgres table under the local runtime migrations?\n- Should entity extraction happen inline in the projector or as a separate enrichment queue?\n- Which node labels are worth first-class treatment in v1 beyond provenance nodes?\n- Should Neo4j vector indexes be ignored entirely in v1, or used later for graph-local reranking?\n\n## Recommendation\n\nProceed with a provenance-first Neo4j sidecar.\n\nDo not start with open-ended entity extraction.\n\nThe first shipping graph should answer:\n\n- where did this memory come from?\n- what source artifacts support it?\n- what is directly connected to it?
